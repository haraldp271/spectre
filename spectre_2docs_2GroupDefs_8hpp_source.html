148 /*!
149  * \defgroup Parallel Parallelization
150  * \brief Functions, classes and documentation related to parallelization and
151  * Charm++
152 
153 SpECTRE builds a layer on top of Charm++ that performs various safety checks and
154 initialization for the user that can otherwise lead to difficult to debug
155 undefined behavior. The central concept is what is called a %Parallel
156 Component. A %Parallel Component is a struct with several type aliases that
157 is used by SpECTRE to set up the Charm++ chares and allowed communication
158 patterns. It might be most natural to think of %Parallel Components
159 as input arguments to a metaprogram that writes the parallelization
160 infrastructure that you requested for the executable. There is no restriction
161 on the number of %Parallel Components, though practically it is best to have
162 around 10 at most.
163 
164 Each %Parallel Component must have the following type aliases:
165 1. `using chare_type` is set to one of `Parallel::Algorithms::Singleton`,
166  `Parallel::Algorithms::Array`, `Parallel::Algorithms::Group`, or
167  `Parallel::Algorithms::Nodegroup`. What these mean is explained below.
168 2. `using metavariables` is set to the Metavariables struct that stores the
169  global metavariables. It is often easiest to have the %Parallel
170  Component struct have a template parameter `Metavariables` that is the
171  global metavariables struct. Examples of this are given below.
172 3. `using action_list` is set to a `typelist` of the Actions that the Algorithm
173  running on the %Parallel Component executes. The Actions are executed in
174  the order that they are listed in in the typelist.
175 4. `using inbox_tag_list` is set to a typelist of the Tags that can be sent to
176  the %Parallel Component by calling the `receive_data` method. If the Inbox
177  %Tag needs to have some temporal locality, then an optional type alias
178  `temporal_id` must be specified in the Inbox %Tag and sent along with the
179  data.
180 5. `using initial_databox` is set to the type of the DataBox that will be passed
181  to the first Action of the `action_list`. Typically it is the output of some
182  `explicit_single_action` call made during the `initialize` function. More
183  on this below.
184 6. `using options` is set to a (possibly empty) typelist of the option structs
185  which are read in from the input file specified in the main `Metavariables`
186  struct and passed to the `initialize` function described below.
187 
188 The following type aliases are not always required, but will be in some
189 circumstances:
190 1. `using array_index` is set to the type that indexes the %Parallel Component
191  Array and is only required if `chare_type = Parallel::Algorithms::Array`.
192  Charm++ allows arrays to be 1 through 6 dimensional or be indexed by a custom
193  type. The Charm++ provided indexes are wrapped as `Parallel::%ArrayIndex1D'
194  through `Parallel::ArrayIndex6D`. When writing custom array indices, the
195  Charm++ manual tells you to write your own `CkArrayIndex`, but we have
196  written a general implementation that provides this functionality; all that
197  you need to provide is a plain-old-data struct of the size of at most 3
198  integers.
199 2. `using explicit_single_actions` is set to a typelist of typelists for Actions
200  that are not part of the algorithm but can be called remotely at any time,
201  similar in spirit to a member function. For each Action, a typelist is
202  specified with the Action as their first element, and the remaining elements
203  are types of the data that is sent to the %Parallel Component and passed into
204  the Action.
205 3. `using reduction_actions_list` is set to a typelist of typelists. The nested
206  typelists must have exactly two elements. The first element is the Action to
207  call, and the second is the type being reduced. For example, the reduced type
208  could be an `int`, a `double`, or a specialization of
209  `Parallel::ReductionData`.
210 
211 %Parallel Components must also have two static member functions with the
212 following signatures:
213 
214 \code
215 static void initialize(
216  Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache, opts...);
217 \endcode
218 
219 and
220 
221 \code
222 static void execute_next_global_actions(
223  const typename metavariables::Phase next_phase,
224  const Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache);
225 \endcode
226 
227 The `initialize` function is called by the Main %Parallel Component when
228 the execution starts and will typically call an explicit single Action
229 to set up the initial state of the Algorithm, similar to what a constructor
230 does for classes. The `initialize` function also receives arguments that
231 are read from the input file and can the be used to initialize the %Parallel
232 Component. For example, the value of an option could be distributed to all
233 members of a %Parallel Component Array, or could be used to control the size
234 of the %Parallel Component Array.
235 
236 The `execute_next_global_actions` function gets run at the end of each
237 %Parallel Phase and determines what the %Parallel Component should do
238 during the next phase. For example, it may simply call `perform_algorithm`,
239 call a series of single Actions, perform a reduction over an Array, or not
240 do anything at all.
241 
242 An example of a singleton %Parallel Component is:
243 
244 \snippet Test_AlgorithmParallel.cpp singleton_parallel_component
245 
246 and similarly of a %Parallel Component Array:
247 
248 \snippet Test_AlgorithmParallel.cpp array_parallel_component
249 
250 Elements are inserted into the Array by using the Charm++ `insert` member
251 function of the CProxy for the array. The `insert` function is is documented in
252 the Charm++ manual. In the above Array example `array_proxy` is a `CProxy` to
253 the `AlgorithmImpl<Parallel::Algorithms::Array, ...>` and so all the
254 documentation for Charm++ array proxies applies. We always create empty
255 Arrays with the constructor and require users to insert however many elements
256 they want and on which cores they want them to be placed. Note that load
257 balancing calls may result in Array elements being moved.
258 
259 There are four types of Algorithms with one Algorithm object per Charm++
260 chare object. The four types of Algorithms are:
261 1. A Parallel::Algorithms::Singleton where there is only one object
262  in the entire execution of the program.
263 2. A Parallel::Algorithms::Array which holds zero or more
264  elements each of which is an object distributed to some core. An array can
265  grow and shrink in size dynamically if need be and can also be bound to
266  another array. A bound array has the same number of elements as
267  the array it is bound to, and elements with the same ID are on the same
268  core. See Charm++'s chare arrays for details.
269 3. A Parallel::Algorithms::Group is an array with
270  one element per core which are not able to be moved around between
271  cores. These are typically useful for gathering data from array elements
272  on their core, and then processing or reducing it. See Charm++'s group
273  chares for details
274 4. A Parallel::Algorithms::Nodegroup, which is similar to a
275  group except that there is one element per node. For Charm++ SMP (shared
276  memory parallelism) builds a node corresponds to the usual definition of a
277  node on a supercomputer. However, for non-SMP builds nodes and cores are
278  equivalent. We ensure that all entry method calls done through the
279  Algorithm's `explicit_single_action` and `receive_data` functions are
280  threadsafe. User controlled threading is possible by calling the non-entry
281  method member function `threaded_single_action`.
282 
283 ### Entry Methods and Remote Function Invocation
284 
285 Charm++ refers to functions that can be called remotely as entry methods.
286 The Parallel Components provide several generic entry methods. These fall
287 into two classes:
288 1. Entry methods that perform a single Action once
289 2. Entry methods that iterate the Actions in the Algorithm's `action_list`.
290  */

339 /*!
340  * \defgroup TestingFramework Testing Framework
341  * \brief Classes, functions, macros, and instructions for developing tests
342  *
343  * \details
344  *
345  * SpECTRE uses the testing framework
346  * [Catch](https://github.com/philsquared/Catch). Catch supports a variety of
347  * different styles of tests including BDD and fixture tests. The file
348  * `cmake/SpectreAddCatchTests.cmake` parses the source files and adds the found
349  * tests to ctest with the correct properties specified by tags and attributes.
350  *
351  * ### Usage
352  *
353  * To run the tests, type `ctest` in the build directory. You can specify
354  * a regex to match the test name using `ctest -R Unit.Blah`, or run all
355  * tests with a certain tag using `ctest -L tag`.
356  *
357  * ### Comparing double-precision results
358  *
359  * To compare two floating-point numbers that may differ by round-off, use the
360  * helper object `approx`. This is an instance of Catch's comparison class
361  * `Approx` in which the relative tolerance for comparisons is set to roughly
362  * \f$10^{-14}\f$ (i.e. `std\:\:numeric_limits<double>\:\:epsilon()*100`).
363  * When possible, we recommend using `approx` for fuzzy comparisons as follows:
364  * \example
365  * \snippet TestFramework.cpp approx_default
366  *
367  * For checks that need more control over the precision (e.g. an algorithm in
368  * which round-off errors accumulate to a higher level), we recommend using
369  * the `approx` helper with a one-time tolerance adjustment. A comment
370  * should explain the reason for the adjustment:
371  * \example
372  * \snippet TestFramework.cpp approx_single_custom
373  *
374  * For tests in which the same precision adjustment is re-used many times, a new
375  * helper object can be created from Catch's `Approx` with a custom precision:
376  * \example
377  * \snippet TestFramework.cpp approx_new_custom
378  *
379  * Note: We provide the `approx` object because Catch's `Approx` defaults to a
380  * very loose tolerance (`std\:\:numeric_limits<float>\:\:epsilon()*100`, or
381  * roughly \f$10^{-5}\f$ relative error), and so is poorly-suited to checking
382  * many numerical algorithms that rely on double-precision accuracy. By
383  * providing a tighter tolerance with `approx`, we avoid having to redefine the
384  * tolerance in every test.
385  *
386  * ### Attributes
387  *
388  * Attributes allow you to modify properties of the test. Attributes are
389  * specified as follows:
390  * \code
391  * // [[TimeOut, 10]]
392  * // [[ErrorRegex, The error message expected from the test]]
393  * SPECTRE_TEST_CASE("Unit.Blah", "[Unit]") {
394  * \endcode
395  *
396  * Available attributes are:
397  *
398  * <table class="doxtable">
399  * <tr>
400  * <th>Attribute </th><th>Description </th>
401  * </tr>
402  * <tr>
403  * <td>TimeOut </td>
404  * <td>override the default timeout and set the timeout to N seconds. This
405  * should be set very sparingly since unit tests are designed to be
406  * short. If your test is too long you should consider testing smaller
407  * portions of the code if possible, or writing an integration test instead.
408  * </td>
409  * </tr>
410  * <tr>
411  * <td>ErrorRegex </td>
412  * <td>
413  * When testing failure modes the exact error message must be tested, not
414  * just that the test failed. Since the string passed is a regular
415  * expression you must escape any regex tokens. For example, to match
416  * "some (word) and" you must specify the string "some \(word\) and".
417  * </td>
418  * </tr>
419  * </table>
420  *
421  * \example
422  * snippet Test_H5.cpp willfail_example_for_dev_doc
423  *
424  * ### Testing static assert
425  *
426  * You are able to test that a `static_assert` is being triggered using
427  * the compilation failure test framework. When creating a new `static_assert`
428  * test you must be sure to not have it in the same file as the runtime tests
429  * since the file will not compile. The new file, say
430  * `Test_StaticAssertDataBox.cpp` must be added to the
431  * `SPECTRE_COMPILATION_TESTS` CMake variable, not `SPECTRE_TESTS`. Here is
432  * an example of how to write a compilation failure test:
433  *
434  * \snippet TestCompilationFramework.cpp compilation_test_example
435  *
436  * Each individual test must be inside an `#%ifdef COMPILATION_TEST_.*` block
437  * and each compilation test `cpp` file must contain
438  * `FILE_IS_COMPILATION_TEST` outside of any `#%ifdef`s and at the end of
439  * the file.
440  *
441  * Specific compiler versions can be specified for which the regex changes.
442  * That is, the compiler version specified and all versions newer than that
443  * will use the regex, until a newer compiler version is specified. For
444  * example, see the below code prints a different static_assert for pre-GCC 6
445  * and GCC 6 and newer.
446  *
447  * \snippet TestCompilationFramework.cpp gnu_versions_example
448  *
449  * ### Debugging Tests in GDB or LLDB
450  *
451  * Several tests fail intentionally at the executable level to test error
452  * handling like ASSERT statements in the code. CTest is aware of which
453  * should fail and passes them. If you want to debug an individual test
454  * in a debugger you need to run a single test
455  * using the %RunTests executable (in dg-charm-build/bin/RunTests) you
456  * must specify the name of the test as the first argument. For example, if you
457  * want to run just the "Unit.Gradient" test you can run
458  * `./bin/RunTests Unit.Gradient`. If you are using a debugger launch the
459  * debugger, for example if you're using LLDB then run `lldb ./bin/RunTests`
460  * and then to run the executable inside the debugger use `run Unit.Gradient`
461  * inside the debugger.
462  */

464 /*!
465  * \defgroup TimeGroup Time
466  * \brief Code related to the representation of time during simulations.
467  *
468  * The time covered by a simulation is divided up into a sequence of
469  * adjacent, non-overlapping (except at endpoints) intervals referred
470  * to as "slabs". The boundaries between slabs can be placed at
471  * arbitrary times. Slabs, as represented in the code as the Slab
472  * class, provide comparison operators comparing slabs agreeing with
473  * the definition as a sequence of intervals. Slabs that do not
474  * jointly belong to any such sequence should not be compared.
475  *
476  * The specific time is represented by the Time class, which encodes
477  * the slab containing the time and the fraction of the slab that has
478  * elapsed as an exact rational. Times are comparable according to
479  * their natural time ordering, except for times belonging to
480  * incomparable slabs.
481  *
482  * Differences in time within a slab are represented as exact
483  * fractions of that slab by the TimeDelta class. TimeDeltas are only
484  * meaningful within a single slab, with the exception that the ratio
485  * of objects with different slabs may be taken, resulting in an
486  * inexact floating-point result. Longer intervals of time are
487  * represented using floating-point values.
488  */
